## High-priority issues

### Bug #H1: Stale `_floor` attribution silently disconnects when build mode changes

- **Item:** 7 (per-floor line-item attribution)
- **Symptom:** A sales rep adds a Zone A line item ("Pooja Room — 80 sqft") and attributes it to "Basement". Later they uncheck `hasBasement` (or change `floors` so the attributed floor no longer exists). The dropdown appears blank (no matching `<option>` for "Basement"), but the underlying `_floor: "Basement"` is still in `state.pricing.zoneLineItems['A'][i]`. The line item's cost still rolls into Zone A's subtotal (good), but its area is silently dropped from the floor summary attribution. If they later re-enable basement, the attribution silently reappears.
- **Root cause:** `app/quote.js`
  - `appendZoneLineItems` line 749: persists `r.floor` blindly.
  - `buildFloorSummary` line 3776: `rows.find(r => r.label === it._floor)` returns `undefined` and bails silently.
  - `renderAreaOverridesPanel` line 2069: `liveFloor = it._floor || ''`; the `<select>` will pick the first `<option>` (the empty placeholder) when no option matches `liveFloor`, so the rep believes the attribution was cleared.
- **Repro:**
  1. Set up a 4-floor stilt + basement build.
  2. Add a Zone A line item, pick floor = "Basement".
  3. Uncheck "Has basement" in the build form.
  4. Reopen the line item — dropdown shows "— floor (optional) —". Floor summary doesn't show 80 sqft anywhere. Save+reload → state still has `_floor: "Basement"`.
- **Suggested fix:** in the dropdown render block (line ~2069), if `liveFloor && !floorOpts.includes(liveFloor)`, render a `<option value="${liveFloor}" selected>${liveFloor} (orphaned)</option>` so the rep can see the stale attribution. OR in `appendZoneLineItems`, drop `_floor` if it's not in `floorOptionsForA(state)`. The former is non-destructive and recoverable; the latter is cleaner but loses data on save.

### Bug #H2: BPF panel still uses generic "Floor 1, Floor 2" labels — nomenclature parity miss

- **Item:** 9 (per-floor balcony pricing format) — adjacent to Item 1 (FLOOR_DISPLAY_NAMES single source of truth)
- **Symptom:** The Per-Floor Balcony Pricing panel labels rows as "Floor 1", "Floor 2", … but everywhere else in the app (calc, summary, line-item dropdown, PDF) they're "Ground Floor", "First Floor", "Second Floor". Inconsistent naming on the very same screen.
- **Root cause:** `app/quote.js` `renderBpfPanel` line 2344: hardcoded `balconyFloorNames = ['Floor 1','Floor 2',...]` array, ignoring `FLOOR_DISPLAY_NAMES`.
- **Repro:** Open the BPF panel after enabling per-floor balcony pricing — compare row labels to the Floor Summary table on the cost page.
- **Suggested fix:** replace `balconyFloorNames` with `FLOOR_DISPLAY_NAMES`. One-line change. Tests don't currently lock this so no test edits needed (though add one in a 7F follow-up).
- **Note:** Item 9 in the brief was scoped to CSS/format parity, not naming, so this is technically a latent bug Item 9 _should have_ caught but wasn't required to. Sales WILL notice this on first use.

## Medium issues

### Bug #M1: ESC doesn't revert inline desc edit (Item 5)

- **Item:** 5 (inline editable description)
- **Symptom:** The brief explicitly asks "Does ESC revert? Does Enter commit? Does blur commit?". Currently: blur commits ✅, Enter does nothing special (form submit prevented elsewhere) ✅, **ESC has no handler** — pressing ESC doesn't revert. The inline RENAME flow (`data-rename-key`) does have an Esc handler (line 2167), so this is inconsistent within the same panel.
- **Root cause:** `renderAreaOverridesPanel` line 2127–2141. The `data-desc-key` input has only `oninput` and `onblur`.
- **Suggested fix:** add `inp.onkeydown = e => { if (e.key === 'Escape') renderAreaOverridesPanel(); };` to mirror the rename handler.

### Bug #M2: Item 5 — no visual cue that an edit is "in flight"

- **Item:** 5
- **Symptom:** While the user is typing, the input is in `descIsOverride=false` styling (italic + muted) on first keystroke, then flips to ink/normal on next render — but the next render only happens on blur. So mid-edit the rep sees italic muted text even though they're typing a custom value. Minor disorientation.
- **Root cause:** styling is computed at render time only. No style change on `oninput`.
- **Suggested fix:** add a CSS rule `input[data-desc-key]:focus { color: var(--ink); font-style: normal; }` so the styling matches the edit state.

### Bug #M3: Stale `it.desc` in area-overrides panel after area override

- **Item:** Cross-cut between Phase 5 (area overrides) and 7E-C Item 5 (desc override)
- **Symptom:** When a user sets an area override for a row, `applyAreaOverrides` (line 911) clobbers `it.desc` to "as per design scope". The Item 5 inline desc input then shows "as per design scope" as the default text & in the placeholder — instead of the original calc formula like "Floor Area (1620) − Staircase (125)". So the rep loses visibility of the original derivation.
- **Root cause:** `applyAreaOverrides` mutates `it.desc` unconditionally; the area-overrides panel reads `it.desc` for the input's default value.
- **Repro:** Enter a build, override "First Floor" area to 2000 → check the description field below it. It now reads "as per design scope" in italic, not the original `Floor Area (1620) − Staircase (125)`.
- **Suggested fix:** snapshot the original desc on first `applyAreaOverrides` pass (mirror what's done with `origName`), and use that as the desc default in `renderAreaOverridesPanel`. OR: don't clobber `it.desc` and let the renderer pick whichever applies based on `it.areaOverridden`.

### Bug #M4: Item 11 floor summary numeric overrides only accept integers, but `buildFloorSummary` parses with `parseFloat`

- **Item:** 11
- **Symptom:** If a user types `1620.5` into a Floor Summary numeric field, the `oninput` handler at line 2444 does `parseInt(t, 10)` → 1620. The decimal silently truncates. But `buildFloorSummary` line 3791 reads back with `parseFloat` (decimals would survive if they got into state). Mild inconsistency; sqft fractions probably never come up in real use.
- **Suggested fix:** either `parseFloat` on input too, or add `step="1"` to the input to make integer-only explicit. Lean toward the latter (sqft is never fractional in practice).

### Bug #M5: Item 11 floor renames don't propagate to the line-item floor dropdown

- **Item:** 7 + 11 cross-interaction
- **Symptom:** User renames "Ground Floor" → "Plinth" in the Floor Summary editor. The summary table now shows "Plinth". But the Zone A line-item floor dropdown (Item 7) still shows "Ground Floor" as a selectable option (because `floorOptionsForA` derives from `FLOOR_DISPLAY_NAMES`, not from the override map). If the rep then attributes a line item to "Ground Floor", `buildFloorSummary` line attribution runs BEFORE the rename pass, so it WORKS — the area gets added to the (still-named "Ground Floor") row, then the rename happens. ✅ math is correct. ❌ but the rep sees "Ground Floor" in one place and "Plinth" in another for the same row.
- **Suggested fix:** in `floorOptionsForA`, if `state.pricing.floorSummaryOverrides[fn[i]]?.label` exists, show the override label as the option text but keep the value as the canonical key. e.g. `<option value="Ground Floor">Plinth</option>`.
