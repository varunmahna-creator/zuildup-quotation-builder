# Phase 7F-B — 2026-05-08 (PDF blank-page elimination)

**Trigger:** Varun on #zuildup-quotation-builder reported on a real client PDF (Davinder Juneja, 211 sqyd, basement+lift+4-floors, 5 additional charges):
1. Cost Calculation heading on its own page; cost table on the next page; the heading page mostly blank.
2. Area Calculation orphan page: just "Continued on next page →" + footer.

7F-A (the same-day Zone-J fix) had moved the symptom but not solved it: forcing the cost table to one page via `break-inside: avoid` produced the "header alone" blank page when total content exceeded one A4.

**Architectural understanding (taped to monitor):**
- `<section.pg>` has `min-height: 297mm` AND `page-break-after: always`.
- If a section's natural content height > 297mm, Chrome paginates: 1 section → 2 (or more) physical PDF pages.
- `pg-foot` is `position: absolute; bottom: 12mm`, so it pins to the bottom of EACH physical page.
- Result: any section that overflows produces a tail-page with little content + the pinned footer = orphan/blank-feeling page.

**Therefore the fix is:** every `<section.pg>` whose content I control MUST stay ≤ 297mm. Period. No more "fix the symptom"; fix the height.

## Three fixes shipped (this commit chain)

**1. Cost section: tbody totals, drop `<tfoot>`, drop `break-inside:avoid`** (commit 4748261)
- `<tfoot>` auto-repeats on every paginated page (CSS `display: table-footer-group`). That was the original "duplicate Sub-total / Construction Total" bug.
- Moving Sub-total / Lift Machine / Construction Total into the LAST `<tbody>` rows with `break-before: avoid-page` keeps them glued to the last zone but makes them appear exactly once.
- Removed the 7F-A `break-inside: avoid` on the table (the strategy that produced the blank "header-only" page).
- New CSS class `cost-totals-sub` / `cost-totals-grand` with the same visual styling as the old tfoot rows.

**2. Area Calc splitPivot tightened** (commit 38ce86b)
- Old budget: `ab <= 9` → splitPivot='AB'. With basement+lift+4-floor: aRows=6, bRows=3, ab=9 → squeezed through but section overflowed by 8mm.
- New budget: `ab <= 6` → splitPivot='A' for any non-trivial build. Page 1 of section now has just Zone A + floor summary; the rest go to the continuation page.
- Also tightened 'ABC' to <=6 and 'A' threshold to <=8 for similar safety.

**3. Floor-summary table padding trimmed** (commit cea2aad)
- Decomposed Area Calc section: floor-summary-block was 82mm — biggest single child.
- thead padding 8px→5px; tbody padding 7px→4px per row; title margin-top 4mm→2mm; block bottom-margin 6mm→3mm.
- Combined saving ≈ 14-17mm.

**4. Continued line margin** (commit 4748261)
- `margin-top: auto` → `margin-top: 6mm`. Stops the line from being pushed to the bottom of the section, which when the section overflows otherwise pushes the line onto its own physical page.

## Lesson (taped to monitor)

When CSS controls print pagination and the layout is print-rendered, **measure section heights in headless Chrome before and after every change**. Don't guess. The 7F-A fix shipped without a height measurement and ended up trading one bug for another; 7F-B was driven by direct getBoundingClientRect data on each child of the failing section, which made it obvious that the floor summary table was the dominant contributor (82mm) and led to the precise CSS trim that solved it.

The 3-commits-on-the-same-bug rule (from SOUL.md) was triggered. Each commit was a different layer of the same architectural problem (section overflow) — not patching the same line, but narrowing the scope based on data. Acceptable because each step had measurement evidence behind it. Going forward, prefer measuring FIRST and committing once with the full understanding rather than the iterate-deploy-measure loop.
