## Commits

### 1. `cffe136` — Phase 7G-A: Save modal — allow renaming saved quote on resave (Bug #4)

**File:** `app/quote.js` (Save modal `saveBtn.onclick`, `save-confirm.onclick`, `QuoteStorage.save()`)

- Save modal now shows the name input PRE-FILLED with the current name even on existing slots, alongside the existing-slot status panel. Reps can rename in place by editing the input and clicking Save.
- `save-confirm.onclick` passes the (possibly edited) name through to `QuoteStorage.save()`. Old behaviour: `save(state)` (no name → blind overwrite). New: `save(state, nameInputVal || undefined)` — empty/whitespace falls back to "no rename" (defensive), non-empty triggers rename.
- `QuoteStorage.save()`: on overwrite (when `state.quoteId` matches an existing index entry), if a non-empty `name` argument is supplied AND it differs from `entry.name`, the entry is renamed in place. The id stays the same — cloud sync semantics unchanged.
- Modal auto-focuses + selects the name input on open in BOTH modes (new and existing) for fast keyboard rename.

**Schema:** unchanged. **Tests:** 174 passed.

### 2. `bf33f98` — Phase 7G-A: Render pending line items so + Add line works (Bug #5)

**File:** `app/quote.js` (`renderAreaOverridesPanel()`)

**Root cause:** the "+ Add line item to Zone X" button pushed a fresh row `{ area: 0, rate: 0 }` into `state.pricing.zoneLineItems[k]` then re-rendered. But the calc engine (`appendZoneLineItems`) skips rows where `area<=0 || rate<=0` because they cannot contribute cost. As a result, freshly added rows were filtered out of `zone.items` and the renderer had nothing to draw — clicks looked like no-ops.

**Fix:** in `renderAreaOverridesPanel`, after rendering `z.items` (the calc-engine view), ALSO iterate `state.pricing.zoneLineItems[k]` and render any rows the calc engine filtered. These render as `.aov-lineitem.aov-lineitem-pending` with a soft-yellow background (`#fffbe6`) so reps see the new row immediately and can fill area/rate inline. Once both are >0, the calc engine picks up the row on the next render and the row "promotes" back into `z.items`.

Existing field-bind handlers (`data-li-field=name|area|rate|desc`, remove ✕, optional Zone-A floor select) work for both pending and promoted rows because they all share the `.aov-lineitem` class.

**Verified live** on the deployed Cloud Run revision via puppeteer: click → pending row appears, fill name+area+rate → promotes to non-pending, persists in `state.pricing.zoneLineItems`.

**Schema:** unchanged. **Calc engine:** unchanged. **Tests:** 174 passed.

### 3. `5b1191c` — Phase 7G-A: Lock entry.customer_name to explicit Save events (Bug #8)

**File:** `app/quote.js` (`QuoteStorage._touch()`)

**Root cause:** `_touch` (the silent overwrite called on every keystroke via `saveState() → auto-persist`) was rewriting `entry.customer_name` from `state.customer.name` on every input event. So a rep who LOADED quote "Hardik — 2026-05-07" and then started editing customer.name to "Davinder Juneja" would see `entry.customer_name` silently morph in the index even without an explicit Save click. Title (`entry.name`) stayed frozen, subtitle (`entry.customer_name`) drifted — they decoupled, and the Load modal showed "Hardik — 2026-05-07" / "Davinder Juneja - by avish - …".

**Fix:** `_touch` no longer touches `entry.customer_name`. The auto-save still persists the slot blob and bumps `modified_at + row_count`, but `customer_name` (and `name`) only update on EXPLICIT save. Title and subtitle now always reflect the same save snapshot.

Pairs cleanly with Bug #4: the rep now consciously confirms both name and customer_name on every Save click.

**Schema:** unchanged. **Tests:** 174 passed.
