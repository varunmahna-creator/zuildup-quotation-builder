## Root cause (from Phase 7N investigation)
Chrome rejected the Inter `data:font/woff2;base64,...` URI inside the full preview HTML with `net::ERR_INVALID_URL`, leaving `document.fonts.get('Inter').status === 'error'`. Likely a Chrome data-URL ceiling (~459 KB encoded for Inter, ~268 KB for Fraunces — only Inter hit the limit). Result: Chrome printed system fallback (LiberationSans) instead of Inter in PDFs. Phase 7N (puppeteer + `document.fonts.ready`) didn't help because the font never finished decoding at all.

## Fix
1. **`app/server.js`**: added public `GET /fonts/:name` endpoint mounted **BEFORE** basic-auth middleware, with strict whitelist (`Inter.var.woff2`, `Fraunces.var.woff2`), `Content-Type: font/woff2`, `Cache-Control: public, max-age=31536000, immutable`. Puppeteer can fetch these without credentials.
2. **`app/server.js`** `renderPdf`: inject `<base href="http://127.0.0.1:${PORT}/">` as the first tag inside `<head>` of the HTML before `page.setContent`. Without this, `page.setContent`'s default base is `about:blank` and `/fonts/...` won't resolve.
3. **`app/quote.js`** `quoteCss()`: replaced the two `@font-face` blocks (~620 KB of inline base64) with HTTP-served `src: url('/fonts/Fraunces.var.woff2') format('woff2-variations')`. Dropped `quote.js` from 988 KB → 244 KB (-744 KB).

## Lessons
- `data:` URIs are not infinite. Chrome has a soft ceiling somewhere around 400+ KB encoded that varies by context. For fonts > 200 KB raw, serve over HTTP from the same origin.
- `page.setContent` defaults base to `about:blank` — always inject `<base href>` if your HTML uses relative URLs.
- Phase 7N's puppeteer migration was still good practice (we now `await document.fonts.ready` properly), but the actual blocker was data URI decode failure, not race conditions.
- Three-way verification (git + revision + end-to-end functional test) is mandatory before "shipped". Spec compliance ≠ functional correctness.
