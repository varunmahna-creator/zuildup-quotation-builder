## The bug Varun reported (2026-05-14 / 15)

> "When user downloads PDF for a quote, the ₹ rupee symbol renders wrong AND fonts (Inter/Fraunces) don't apply — looks like default browser fonts instead of the branded typography."

---

## Root Cause

Phase 7L (revision `00043-6tv`) inlined **three** `@font-face` base64 blocks inside `quoteCss()`:

| Block | Family   | Subset      | Glyphs | Has ₹ (U+20B9)? |
|-------|----------|-------------|--------|-----------------|
| 0     | Fraunces | latin-ext   | 289    | ✅ YES          |
| 1     | Fraunces | latin       | 222    | ❌ **NO**       |
| 2     | Inter    | (subset)    | 96     | ✅ YES          |

Plus `unicode-range` declarations to split which subset handles which codepoint range. On paper, U+20B9 falls into `U+20AD-20C0` (Fraunces latin-ext block), so ₹ *should* have rendered in Fraunces-styled text.

**In practice**, Chrome `--print-to-pdf` did not consistently honour the `unicode-range` switch across the two Fraunces subsets — when ₹ appeared inside a Fraunces-styled element (e.g. `cover-name`, `h1.section`, `floor-summary-title`), Chrome silently fell back to a system serif (Liberation Serif) for that single glyph. Result:

- ₹ glyph metrics + style suddenly diverged from the surrounding Fraunces text → looked "wrong" and broke headline kerning
- In viewers that render Type3 differently from Type0, the fallback was even more visually jarring → "looks like default browser fonts"

The `app/fonts/Fraunces-{400,600,700}.woff2` files Phase 7L shipped on disk were **also** the latin-only subset (no ₹) — verified with `fontTools.ttLib.TTFont.getBestCmap()`:
```
Fraunces-400.woff2: glyphs=222 ₹=False  ← shipped without ₹
Fraunces-600.woff2: glyphs=222 ₹=False
Fraunces-700.woff2: glyphs=222 ₹=False
Inter.var.woff2:    glyphs=2852 ₹=True
```

---

## The Fix

Replace the 3-block (latin + latin-ext + Inter) `@font-face` setup with **two single-block declarations**, each backed by a **full-coverage variable woff2**:

- **Fraunces.var.woff2** (205 532 B, 637 glyphs) — downloaded from upstream
  `github.com/undercasetype/Fraunces/raw/master/fonts/variable/Fraunces[...].ttf`, converted to woff2
  via `fontTools` (`TTFont.flavor='woff2'`). Includes ₹, €, full Latin + Latin-Ext.
- **Inter.var.woff2** (352 240 B, 2 852 glyphs) — unchanged from Phase 7L (Inter v4 variable, full coverage).

Both are embedded as `data:font/woff2;base64,…` inside `quoteCss()`. No `unicode-range` (so no subset gambit). `font-display: block` → `swap` (recommended for print → Chrome more aggressively reuses the loaded font instead of leaving it half-Type3, half-fallback).

Also:
- Removed the now-redundant Google Fonts `@import` from `app/preview.html` (network dep, defeated by base64 fonts anyway).
- Deleted `app/fonts/Fraunces-{400,600,700}.woff2` (none of them referenced anywhere; the base64-in-quoteCss is the only PDF-time font source).

## Operational Lessons / Gotchas

1. **`unicode-range` font subsetting is fragile in Chrome `--print-to-pdf`.** Even when the math says ₹ (U+20B9) should fall into a given subset's `U+20AD-20C0` range, Chrome may not consistently use that font for the glyph. Mitigation: ship **one** full-coverage font per family. The bandwidth cost (≈ 100 KB extra) is a worthwhile trade for deterministic rendering.

2. **Verify .woff2 glyph coverage with `fontTools.ttLib.TTFont.getBestCmap()`** before shipping. The Phase 7L Fraunces files looked correct in metadata (FontFamily="Fraunces") but were missing U+20B9.

3. **The FS-race quirk on this VM is real.** During this session, a `cd /opt/openclaw/workspace/zuildup/quotation-builder` failed silently (directory "didn't exist" for ~5 s) and a subsequent `git add` ran against the wrong repo (root workspace, hundreds of unrelated untracked files). Always re-run `cd && pwd` after a long-running command, especially after writes/deploys.

4. **`gcloud run deploy --source` deploy time** for this Dockerfile + Buildpacks is ~6–7 minutes from start to "100 % traffic". Plan for that. Don't `nohup` it (see Phase 7L lesson #2).

5. **Type3 vs Type0 in Chrome's printed PDF**: When a font is loaded from a `data:` URL, Chrome subsets it into a Type3 PDF font (vector ops embedded as PostScript-like instructions) rather than a Type0/TrueType embed. **The glyphs drawn are still from the loaded font** — verify with `xref_object()` on the FontDescriptor's `/FontFamily`, NOT via `pdffonts -t`. Visual rendering is correct in all major PDF viewers (Adobe Reader, Preview, Chrome's own viewer, mupdf). `font-display: swap` does nudge Chrome to prefer Type0 in some cases but is not a hard guarantee.
