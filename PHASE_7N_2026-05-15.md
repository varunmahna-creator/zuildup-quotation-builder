## Root-cause investigation (deeper than the task brief assumed)

I ran puppeteer-core locally against the same preview HTML and inspected `document.fonts`:

```json
{
  "fonts": [
    { "family": "Fraunces", "status": "loaded", "weight": "100 900" },
    { "family": "Inter",    "status": "error",  "weight": "100 900" }
  ],
  "bodyFont": "Inter, system-ui, sans-serif"
}
```

Inter's `FontFace.status` is `"error"` — Chrome's font loader **rejected the data URI entirely**, before any rendering happened. Page console emitted:

```
REQFAIL: data:font/woff2;base64,d09GMgABAAAABV/wABMAAAANdbgABV91AAQAQgAA...   net::ERR_INVALID_URL
PAGE: error Failed to load resource: net::ERR_INVALID_URL
```

This is **not a font-decode race**. It is **Chrome refusing to parse the Inter data URI as a valid URL** when delivered inside the full preview HTML. The same Inter @font-face block in a **minimal standalone HTML** loads correctly (Inter status = "loaded"). Something about the surrounding context in `quote.js`'s preview HTML triggers Chrome's data-URL rejector — most likely:

1. **Combined embedded-font size of the page** — full preview is ~1.1 MB and contains two large fonts inline (Inter 459 KB b64 + Fraunces 268 KB b64). Chrome 148 may have an aggregated data-URL ceiling per stylesheet.
2. **Subtle parsing edge case** in the data URI when followed by additional CSS inside the same `<style>` block.

This bug is **independent of the `--print-to-pdf` -> puppeteer migration**. Even with the (correct) puppeteer code path and the (correct) `document.fonts.ready` wait, the font never gets parsed because Chrome rejects the URI at the network layer.

## Recommended Phase 7O (next step)

Two paths to actually fix Inter rendering:

**Option A — serve fonts over HTTP instead of data URI (RECOMMENDED).**
Expose `/fonts/inter.woff2` and `/fonts/fraunces.woff2` as static endpoints (or `app.get` handlers) on the same Express app; change `quote.js` `quoteCss()` to `url('/fonts/inter.woff2')`. Then in `renderPdf`, instead of `setContent(html)`, we'd need either:

- An ephemeral local HTTP server (puppeteer can hit `http://localhost:PORT/...`), OR
- `page.setRequestInterception(true)` to intercept the font requests and serve them from disk.

Chrome handles network URLs reliably regardless of size; data URL parser limits are bypassed entirely.

**Option B — bisect the data URI rejection.** Split the two @font-face blocks into separate `<style>` tags, or reduce Inter to a non-variable subset (only weight 400/600/700) to shrink the base64. Less robust.

My recommendation: **Option A**, because it's robust and matches modern web font delivery patterns. The data URI approach was clever but hits a Chrome limit.
