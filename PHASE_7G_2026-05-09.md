## Phase 7G-A — Save / Load / Add-line bug-fix sweep

Owner: dedicated subagent. Scope: items #4, #5, #8. Detailed write-up: see `PHASE_7G-A_2026-05-09.md` in this directory. Highlights below.

### `cffe136` — Bug #4: editable quote name on resave
- Save modal now PRE-FILLS the name input on existing slots (previously hidden, forcing Save-As-Copy).
- `save-confirm` passes the (possibly edited) name to `QuoteStorage.save(state, nameInputVal || undefined)`.
- `QuoteStorage.save()`: on overwrite, if name argument differs from current `entry.name`, the entry renames in place (id unchanged → cloud sync semantics preserved).
- Modal auto-focuses + selects the name input on open for fast keyboard rename.

### `bf33f98` — Bug #5: + Add line worked, just didn't render
- Root cause: clicking `+ Add line item to Zone X` correctly pushed `{ area: 0, rate: 0 }` into `state.pricing.zoneLineItems[k]`, but the calc engine `appendZoneLineItems` filters `area<=0 || rate<=0` rows since they can't contribute cost. The renderer iterated `z.items` (calc-engine view), found nothing, drew nothing.
- Fix: `renderAreaOverridesPanel` now ALSO iterates `state.pricing.zoneLineItems[k]` and renders any rows the calc engine filtered. Pending rows render with `.aov-lineitem.aov-lineitem-pending` class and a soft-yellow background (`#fffbe6`) so reps see them immediately. Once area + rate > 0, the row "promotes" back into `z.items`.
- Verified live via puppeteer on the deployed Cloud Run revision.

### `5b1191c` — Bug #8: customer_name drift on auto-save
- Root cause: `QuoteStorage._touch()` (the silent auto-persist called on every keystroke) was rewriting `entry.customer_name` from `state.customer.name`. Title (`entry.name`) stayed frozen at first explicit Save, subtitle (`entry.customer_name`) drifted on every keystroke → they decoupled. The screenshot Varun shared shows exactly this: title "Hardik — 2026-05-07" with subtitle "Davinder Juneja - by avish - …".
- Fix: `_touch` no longer touches `entry.customer_name` (or `entry.name`). Both fields update ONLY on explicit Save. Pairs cleanly with Bug #4 — reps now consciously confirm both on every Save click.

### Known limitation (Bug #8)
- Existing already-corrupted entries do NOT auto-heal. The fix prevents NEW drift. If Varun wants historical cleanup, a one-off pass to sync `entry.customer_name = slot.customer.name` is possible — flagged as a follow-up.

### `b36fcc1` — #3: no-stilt ground-floor logic

For `buildType === 'nostilt'`, the GROUND FLOOR (i=0) row in the area calculation now:
- Has **NO balcony** — both in the floor-summary semi-covered column AND in the Zone B per-floor balcony expansion AND in the combined balcony total.
- **Open Area = setback + ramp** — pulled post-override from the Zone C item set (with a fallback to `(plotSqFt - floorArea) / breadth * RAMP_DEPTH` if those zone items are missing).

Stilt and structure-only paths are unchanged (regression-tested via the new JS unit test below). This matches the semantic Varun described: in nostilt builds, the ground floor sits directly on the plot — no stilt slab below, no cantilevered balcony jutting over an open ground area. Setback + ramp is what's actually open.

### `f40baaa` — #9: areas display with 2 decimal places

Varun's mid-session addition. Previously all area values were rounded to integers (`Math.round`, `toFixed(0)`, or via the `ni()` Indian-number formatter which is integer-only).

**Change:**
- New helper `niA(n)` = `Number(n).toLocaleString('en-IN', { minimumFractionDigits: 2, maximumFractionDigits: 2 })`. Produces `1,556.50` from `1556.5`, `1,736.00` from `1736`.
- Removed `Math.round` calls on `floorArea` in `calcPackage`, `calcStructure`, `buildFloorSummary` — non-integer areas now flow end-to-end without precision loss.
- Replaced `ni()` → `niA()` at AREA render sites only: cover-page subtitle (Plot Area sqyd/sqft), Area Calculation description + table cells, Floor Summary cells, Total Built-up Area row.
- **Untouched:** money formatting (`fmtINR`), liter volumes (water tank Zone D — still `ni()` since liters are integer counts of L), floor count, lift stops, anything else that's a cardinal integer.

### `4bcae34` — #6: brand field on spec line items

**Schema (additive, migration-safe):**
- Every spec line-item override now has an optional `brand` field (string, default empty).
- Editor UI: dedicated Brand input sits between Rate and Description.
- PDF + HTML render: brand line rendered in **bold ABOVE the description** in BOTH grid mode (`.spec-card`) and table mode (`.spec-table-block`).
- If `brand` is blank: NO brand line rendered. No "Brand: —" placeholder, nothing.

**Migration:**
- Legacy saved quotes don't have the `brand` key. On load, the key falls through to `BRAND_DEFAULTS` — a map keyed by catalog item id, populated from `_hardik_reference_quote.pdf` (Asian Paints, Jaquar, Crompton, Otis, Sika, Ozone, Godrej, Tractor, etc.).
- A DELIBERATELY-blanked empty STRING (rep explicitly cleared the field) suppresses the brand line — distinguished from "key absent" so the rep can opt out per item without the default reappearing.

### `f0afce9` + `e8cf0cd` — #7: FAR API integration

**Status:** UI + JS plumbing fully wired. The actual live FAR call depends on the FAR API service being deployed and reachable at the URL configured in `FAR_API_URL` (Cloud Run env var on `zuildup-quotes`). The integration degrades gracefully when the API is unreachable — sales team sees a clear "FAR API unavailable — enter manually" hint and the quote remains usable.

**HTML changes (`app/index.html`):**
- New `<small id="far-hint">` under the Coverage (%) input. Default text: "Auto-populated from FAR API. Edit to override."
- New `<button id="far-reset">↻ Reset to FAR default</button>` next to the input.

**JS changes (`app/quote.js`):**
- `_doFarFetch(state)` — debounced (500ms) call to `FAR_API_URL` with `{ length_ft, breadth_ft, location }` payload. On success: writes to `state.coverage` if `state.coverageOverridden !== true`; updates `#far-hint` text with the returned value. On failure: shows the "API unavailable" hint and surfaces no error to the rep.
- `f-coverage` input listener: if the rep edits the field manually, sets `state.coverageOverridden = true` and changes hint text to "Manual override. ↻ to reset."
- `#far-reset` click handler (added in `e8cf0cd` after being lost during the concurrent-edit window): clears `coverageOverridden`, re-triggers `_doFarFetch`, restores auto-populated value.
- Reactive triggers: plot-length, plot-breadth, location field changes all kick a debounced FAR fetch.

**Known issue (from 7G-A doc, surfaced during puppeteer verification):**
- `[PAGEERR] state is not defined at _doFarFetch (quote.js:716)` — closure-scope bug. The `_doFarFetch` function references `state` but the variable isn't in scope at that call site. Currently the function silently throws; the catch path shows the "FAR API unavailable" hint, so the rep sees graceful fallback rather than a broken page. **Follow-up needed** to wire the state correctly through the closure or refactor `_doFarFetch` to accept state as an argument.

## Phase 7G-D — LLM tier-pre-fill (PARKED)

### Scope (when training data lands)
1. Sales rep clicks "New Quote", system asks: which tier — luxury / mid-luxury / premium / budget?
2. Based on selection, ALL rates, descriptions, and brand defaults populate from the tier template.
3. Rep edits inline via natural-language prompts ("change all bathroom fittings to Kohler", "drop steel rate by 5%", etc.) instead of clicking through every field.
4. System learns from edits — every saved quote feeds back into the tier templates, refining defaults over time.

### Architecture (proposed, not yet committed)
- **Tier templates:** stored in `catalog/tiers/{luxury,mid_luxury,premium,budget}.json`. Each template = full quote skeleton (rates per zone + spec line items + brands + descriptions).
- **Pre-fill flow:** "New Quote" modal adds a tier dropdown. On selection, deep-merge tier template into fresh state.
- **LLM edit flow:** small prompt box in the editor. Rep types natural language → LLM translates to a JSON diff against current state → diff applied via existing state-mutation pathways. Use Anthropic Sonnet (cost: cheaper than Opus, ample for this kind of structured edit).
- **Learning loop:** on every Save, run a Sonnet pass that compares the saved quote to the tier template the rep started from. If a particular field (e.g. cement brand) has drifted away from the template in ≥3 saved quotes of the same tier, surface a "Update tier default?" prompt to Varun (2-eyes rule). Approved updates merge back into the tier template.
- **Storage:** Cloud SQL `zuildup-db` (existing instance, asia-south1) with new tables `tier_templates`, `tier_edits_log`, `tier_default_suggestions`.

### Blocking on Varun
- Multi-tier sample quotes labelled by tier + price band (Varun committed to send these). Without them, the tier templates are guesswork.
- Decision on how to expose the LLM edit prompt: inline per-field, global prompt box, or both?
